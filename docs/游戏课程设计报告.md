## 摘要

本报告阐述了一款基于 C++17 和 SFML 3.0 图形库开发的五子棋卡牌策略游戏（ChessBoardGame）的设计与实现过程。该游戏在传统五子棋规则基础上，创新性地引入了卡牌对战机制，玩家通过落子形成三连/四连模式来抽取卡牌，利用卡牌效果改变战局。游戏实现了完整的行动点（AP）系统、6 张橙卡 + 2 张紫卡的差异化效果、三级 AI 对战、以及一套基于预计算碎片缓存的高性能碎片动画系统——涵盖棋子和卡牌的生成、销毁、转化、聚合、湮灭全过程的粒子级视觉效果。报告从需求分析、系统设计、核心技术实现、测试验证等方面全面阐述了项目的开发过程。

**关键词**：五子棋；卡牌策略；SFML；碎片动画；C++17；游戏开发

## 一、项目概述

### 1.1 项目背景

五子棋作为中国传统的棋类游戏，规则简洁、受众广泛，但单一的下棋玩法容易导致对局重复性高、策略深度有限。近年来，《炉石传说》《杀戮尖塔》等将卡牌机制与传统玩法融合的游戏取得了巨大成功，证明了"棋类 + 卡牌"这一混合品类的巨大潜力。

基于此背景，本团队设计并实现了一款融合五子棋与卡牌策略玩法的双人对战/人机对战游戏——ChessBoardGame。游戏在保留五子棋经典规则的基础上，增加了卡牌收集、策略出牌、行动点管理等机制，丰富了游戏的策略层次和可玩性。

### 1.2 项目目标

1. 实现完整的 15×15 五子棋棋盘系统，支持黑/白双方轮流落子与独立胜利条件判定
2. 构建创新性的卡牌系统，包含橙卡（正面效果卡）和紫卡（负面诅咒卡）两大类
3. 设计行动点（AP）框架，将下棋与出牌纳入统一的资源管理系统
4. 实现三级 AI 难度的人机对战模式
5. 开发高性能碎片动画系统，实现棋子销毁/转化、卡牌生成/湮灭的粒子级视觉效果
6. 提供完整的 UI 界面，包括主菜单、对战配置、暂停菜单、对局复盘等

### 1.3 项目特色

- **策略深度融合**：棋类与卡牌两种玩法通过行动点系统有机统一，每一步都需要权衡下棋还是出牌
- **创新的胜利条件**：摒弃传统的"一局定胜负"，改为集棋子获胜制——每达成一次五连/六连，在 ChessFrame 装饰框上放置一枚己方棋子，率先集满 5 枚者获胜
- **碎片动画系统**：棋子和卡牌均支持全流程的碎片化视觉特效，包括生成（聚合）、销毁（粉碎）、转化（粉碎+聚合）、湮灭（上升粉碎）等效果
- **紫卡机制**：引入"敌对传送"概念，紫卡不是自己使用而是传送给对手，实现间接博弈

## 二、系统需求分析

### 2.1 功能性需求

#### 2.1.1 棋盘系统
- 15×15 标准棋盘渲染
- 黑白双方轮流落子，支持鼠标点击操作
- 每方独立胜利条件（默认五子连星，可通过卡牌效果修改为六子）
- 三连/四连模式检测与抽卡触发
- 胜利连线高亮显示与集棋子获胜制（ChessFrame 系统）
- 落子历史记录与复盘回放

#### 2.1.2 行动点系统（AP）
- 三种行动点模式：双能点（可下棋+可出牌）、仅下棋、仅出牌
- 智能消耗算法：优先消耗专能点，保留双能点
- AP 池结算：归零切换回合，未归零则锁定时钟继续行动
- 卡牌效果可赠送额外行动点（如"连击"卡赠送 2 次落子机会）

#### 2.1.3 卡牌系统
- **牌库管理**：随机抽卡机制，共享牌库，支持已用卡牌放回
- **橙卡出牌**：拖拽至 CardReader 区域 → 碎片湮灭动画 → 中央展示 → 触发效果
- **紫卡传送**：拖拽至 CardPortal 区域 → 下滑传送动画 → 加入敌方手牌
- **双卡槽系统**：支持多张手牌堆叠存放，独立拾取
- **卡牌种类**（共 9 张）：

| 卡牌 | 类型 | 效果 |
|------|:---:|------|
| 连击 | 橙 | 赠送 2 次额外落子机会 |
| 隐忍 | 橙 | 敌方胜利条件改为六子连星 |
| 笼络 | 橙 | 销毁己方一子 + 转化敌方一子 |
| 破釜沉舟 | 橙 | 手牌全部退库，按数量随机销毁敌方棋子 |
| 疫病 | 橙 | 感染敌方棋子，每回合概率死亡并向邻近传播 |
| 隔离 | 橙 | 三回合后清除全部疫病 |
| 悬崖勒马 | 橙 | 回退到上一个己方回合（悔棋） |
| 盲目 | 紫 | 手牌名称和描述乱码，三回合消退 |
| 以地事秦 | 紫 | 持有者需传橙卡回应，否则三回合禁止出牌 |

#### 2.1.4 AI 系统
- 三级难度：初级、明智、大神
- AI 自动检测三连/四连 → 抽卡
- 基于启发式评估的卡牌选择
- 完整动画流程：Portal 生成 → 展示 → 滑入 Reader → 湮灭 → 效果触发
- 紫卡优先出牌策略

#### 2.1.5 UI 与交互
- 主菜单（星空背景 + PIP 动画 + 可拖拽乱码卡片）
- 对战配置界面
- 对局 HUD（计时器、卡槽、ChessFrame 装饰框、悬停预览）
- 暂停菜单（ESC 切换，半透明遮罩 + 下落动画）
- GAME_OVER 界面（胜负提示 + 返回主菜单）
- F10 调试面板（AI 手牌查看、卡牌即时生成）

```
│  GameEngine  │───▶│  Chessboard  │    │  DeckManager │

│   事件分发)  │    │   碎片动画)  │    │              │
└──────┬───────┘    └──────────────┘    └──────────────┘
       │
       ▼
┌──────────────┐    ┌──────────────┐
│   AIPlayer   │    │    Card.h    │
│  (AI决策/    │    │  (卡牌结构/  │
│   难度控制)  │    │   效果枚举)  │
└──────────────┘    └──────────────┘
```

**核心类职责**：
- **GameEngine**（~6000 行）：游戏主控类，管理状态机（MENU → PVE_CONFIG → GAME_PVP/PVE → GAME_OVER）、行动点系统、事件处理、渲染调度、卡牌效果分发、碎片动画渲染、音效管理、UI 渲染
- **Chessboard**（~1200 行）：棋盘逻辑，包括网格管理、落子判定、胜利检测（四方向扫描）、三连/四连模式检测、棋子碎片动画（销毁/转化/衰减）
- **DeckManager**（~60 行）：牌库管理，负责初始牌库构建、随机抽卡、手牌管理、AI 手牌管理
- **AIPlayer**（~600 行）：AI 决策系统，包括落子评估、卡牌选择、启发式评分
- **Card.h**：数据结构层，定义 `Card` 结构体和 `CardEffect` 枚举

### 3.3 行动点系统设计

行动点（Action Point）是本游戏"棋牌融合"的核心纽带。每回合开始玩家获得一个双能点（既可下棋又可出牌）。消耗时采用**智能扣减策略**：

```
优先级：专能点（仅下棋/仅出牌）> 双能点

例如：
- 持有 [下棋+出牌, 仅下棋]，下棋时优先消耗 [仅下棋]，保留 [下棋+出牌] 以便后续出牌
- 持有 [下棋+出牌]，下棋后保留 [仅出牌]
```

每次行动后执行总体结算（`settleActionPoints()`），AP 池为空时切换回合，非空时锁定倒计时继续行动。这一设计使得快节奏的"连击"效果（如连击卡赠送 2 次落子点）自然融入系统。

### 3.4 数据流设计

**出牌流程**：
```
玩家拖牌到 CardReader
  → 碰撞检测（圆心半径判定）
  → AP 检查（hasValidActionPoint）
  → 5 阶段湮灭动画（PAUSE_BEFORE → MOVE_TO_200 → PAUSE_AFTER → SHATTER）
  → consumeActionPoint
  → 中央展示（PAUSE → APPEAR → DISPLAY → FADE_OUT）
  → applyCardEffect → settleActionPoints
```

**紫卡传送流程**：
```
玩家拖牌到 CardPortal
  → 碰撞检测
  → MOVE_TO_POS → PAUSE → MOVE_TO_PORTAL（下滑）
  → 卡牌标记 transmitted=true
  → 插入敌方手牌
  → 敌方回合时自动生效
```

## 四、核心技术实现

### 4.1 碎片动画系统（重点）

碎片动画系统是本游戏视觉效果的核心，也是指导老师重点关注的部分。该系统实现了一套完整的**预计算 + 运行时物理模拟**的粒子动画框架，覆盖棋子和卡牌的生成、销毁、转化、湮灭全过程。

#### 4.1.1 Fragment 数据结构

所有碎片动画共享统一的 `Fragment` 结构体（定义于 `Chessboard.h`）：

```cpp
struct Fragment {
    sf::Vector2f pos;           // 当前世界坐标
    sf::Vector2f vel;           // 速度 (px/s)
    sf::IntRect  texRect;       // 源纹理子区域（定义该碎片对应纹理的哪一块）
    float rotation = 0.f;       // 当前旋转角度 (度)
    float rotSpeed = 0.f;       // 旋转速度 (度/s)
    float alpha = 255.f;        // 透明度 (0-255)
    bool  released = false;     // 擦除线是否已越过该碎片（触发释放）
    sf::Vector2f targetPos;     // 聚合目标位置（聚合动画使用）
    float assembleTimer = 0.f;  // 聚合计时器
    float fadeTimer = 0.f;      // 淡出计时器（smoothstep 曲线）
};
```

该结构体同时支持**粉碎动画**（从完整位置散落消失）和**聚合动画**（从散落位置聚合到目标），通过 `released` 标志区分初始状态，通过 `targetPos` 指定聚合目标。

#### 4.1.2 碎片预计算缓存

为优化运行时性能，所有碎片图案在游戏启动时**一次性预计算**并缓存，运行时直接复制缓存进行物理更新。

**棋子碎片缓存**（`Chessboard::initFragmentCache()`，12×12=144 块）：

```cpp
auto genFragments = [](sf::Texture& tex, std::vector<Fragment>& out) {
    const int COLS = 12, ROWS = 12;
    float cw = (float)ts.x / COLS;   // 每格宽度
    float rh = (float)ts.y / ROWS;   // 每格高度
    for (int r = 0; r < ROWS; ++r) {
        for (int c = 0; c < COLS; ++c) {
            Fragment f;
            // 在网格基础上添加随机扰动，使碎片边缘不规则
            int rx = (int)(cw * c + cw * (rand() % 16 - 8) / 100.f);   // X ±8%
            int ry = (int)(rh * r + rh * (rand() % 8  - 4) / 100.f);   // Y ±4%
            int rw = (int)(cw + cw * (rand() % 14 - 7) / 100.f);        // 宽 ±7%
            int rh2 = (int)(rh + rh * (rand() % 8  - 4) / 100.f);       // 高 ±4%
            // 夹紧到纹理边界
            f.texRect = sf::IntRect({rx, ry}, {rw, rh2});
            out.push_back(f);
        }
    }
};
```

**卡牌碎片缓存**（`GameEngine::initCardFragmentCache()`，36×36=1296 块）：原理相同，网格更密集。同时为橙卡（`Card.png`）和紫卡（`Card_01.png`）分别生成缓存，以及为 ChessFrame 动画生成 black_HR/white_HR 的 36×36 碎片缓存（`initFramePieceFragCache()`）。

**设计要点**：
- Y 方向扰动收紧（±4% vs X 方向 ±8%），确保自上而下的擦除线保持清晰
- 使用百分比扰动而非像素绝对值，使纹理尺寸变化时效果一致
- 一次性预计算避免了每帧大量随机数生成和纹理分割的开销

#### 4.1.3 棋子销毁动画（粉碎，Fragment Destruction）

棋子销毁时（如笼络卡销毁己方棋子、破釜沉舟批量销毁），使用自下往上的擦除线扫描，碎片释放后受重力影响坠落消散。

**核心逻辑**（`Chessboard::draw()` 内联渲染）：

```
1. 擦除线计算：
   clipHeight = textureHeight × (1 - t)    // t 从 0→1，擦除线从底到顶

2. 未粉碎部分（clipHeight 以上）：
   绘制完整裁剪纹理，无碎片化 → 视觉上保持棋子完整性

3. 已粉碎部分（clipHeight 以下）：
   遍历 144 个碎片：
   if 碎片纹理位置 > clipHeight:
       碎片.released = true
       初始化物理参数（重力 800px/s²、水平漂移 ±80px/s、旋转 ±360°/s）
   
   if 碎片.released:
       物理更新（欧拉积分）:
         vel.y += 800 * dt
         pos += vel * dt
         rotation += rotSpeed * dt
       透明度更新（smoothstep 淡出，0.85s）:
         alpha = 255 × (1 - t³(3-2t))
       绘制碎片（带 BlendAdd 发光层叠）
```

**关键物理参数**：
- 重力加速度：800 px/s²（模拟真实坠落感）
- 水平漂移速度：±80 px/s（随机，模拟爆炸飞散）
- 旋转速度：±360 °/s（随机，增加动感）
- 淡出时长：0.85s（smoothstep 曲线，缓入缓出）

**衰减缓冲区**（`decayFragments`）：动画结束后，残留碎片继续播放物理 + 淡出，直至 `alpha ≤ 0` 自动移除，避免动画突然消失的突兀感。

#### 4.1.4 棋子转化动画（粉碎 + 聚合，Fragment Conversion）

笼络卡第二阶段的转化效果，同时运行两套碎片系统：

```
敌方棋子（fromColor）：
  碎片从棋子位置粉碎坠落（同销毁动画）

我方棋子（toColor）：
  碎片从棋子下方区域（随机偏移）飞向目标位置聚合
  pos += (targetPos - pos) × 0.12f    // 指数平滑收敛
  alpha 从 0 淡入至 255               // smoothstep 缓入
```

两种动画同时渲染，视觉上呈现"旧棋子粉碎、新棋子生长"的效果。

#### 4.1.5 卡牌湮灭动画（上升粉碎，Card Annihilation）

橙卡出牌时，卡牌滑入 CardReader 后边上升边粉碎（`CardAnnihilateState::SHATTER`）：

```
初始位置：(300, 540) → 最终位置：(300, 210)
上升过程中擦除线自下往上扫描：
  未粉碎部分 → 裁剪纹理渲染（保持完整卡牌的上半部分）
  已粉碎部分 → 碎片散落，无重力（边上升边自然飘散）
```

此设计与棋子销毁的"坠落"效果形成反差——卡牌向上湮灭突出"消耗/献祭"感。

#### 4.1.6 卡牌出生/展示聚合动画（Fragment Aggregation）

卡牌生成时（抽牌、中央展示 APPEAR 阶段、帧棋子动画 AGGREGATE 阶段），使用自上往下的擦除线扫描，碎片从散落位置飞向目标聚合。

**核心逻辑**：

```
1. 初始离散化：
   复制碎片缓存，随机化每个碎片的起始位置和旋转
   for each 碎片:
     pos = 目标中心 + 随机偏移 (±150px X, +80~180px Y)

2. 擦除线扫描（自上往下）：
   wipeY = textureHeight × t    // t 从 0→1

3. 已聚合部分（wipeY 以上）：
   绘制完整裁剪纹理（无缝衔接）

4. 聚合中碎片（wipeY 以下）：
   if 碎片纹理位置进入释放窗口 [wipeY-2%~wipeY+6%]:
     释放，设 targetPos，将 pos snap 到目标附近
   
   if 碎片.已释放:
     pos += (targetPos - pos) × 0.12f     // 指数平滑追踪
     alpha = 255 × smoothstep(t/0.3s)     // 0.3s 缓入
     旋转衰减：rotation × (1 - eased)
     叠加 BlendAdd 发光（飞行中渐亮，到达后缓暗）
```

**释放窗口机制**：碎片在擦除线 ±2%~6% 的范围内触发释放，而非精确匹配。这一设计避免了"所有碎片同时释放"的僵硬感，产生自然的渐进式聚合效果。

#### 4.1.7 帧棋子动画（ChessFrame Piece Animation）

本游戏最新实现的 ChessFrame 集棋子获胜制配套动画，融合了聚合和飞行两种效果：

```
AGGREGATE (1.0s): 中央碎片聚合，形成大 HR 图 (~384px)
HOLD (0.5s):      静止展示
FLY (0.5s):       smoothstep 飞行至 ChessFrame 目标槽位
                  缩放：384px → 96px
                  末帧切换为普通棋子纹理
```

该动画复用了卡牌展示 APPEAR 的全部碎片渲染逻辑，额外增加了 smoothstep 飞行缩放阶段，体现了系统的可复用性。

### 4.2 行动点（AP）系统实现

行动点系统以 `std::vector<ActionPoint> currentTurnActionPoints` 作为存储容器：

```cpp
struct ActionPoint {
    bool canPieceDrop;  // 允许下棋
    bool canPlayCard;   // 允许出牌
};
```

**智能消耗算法**（`consumeActionPoint(bool isPieceDrop)`）：
1. 收到消耗请求后，先遍历查找是否有对应的专能点
2. 若找到，消耗该专能点
3. 若未找到，消耗一个双能点（将对应标志位置为 false）
4. **优先保留双能点**：若存在 [仅下棋] + [下棋+出牌]，要求下棋时会优先消耗 [仅下棋]

**总体结算**（`settleActionPoints()`）：
1. 移除所有 `canPieceDrop==false && canPlayCard==false` 的无效点
2. 若 AP 池为空 → 回合结束，切换 `currentTurn`，重置 45s 倒计时
3. 若 AP 池非空 → 继续行动，倒计时冻结不重置

### 4.3 胜利检测与集棋子获胜制

**连线检测**（`Chessboard::checkWin()`）：从落子位置出发，向四个方向（水平、垂直、对角线、反对角线）扫描，统计连续同色棋子数，与 `winCondition[player]` 比较（默认 5，隐忍卡可修改为 6）。

**scoredWin 标记系统**：为防止同一条五连线被重复计分，引入 `scoredWin[15][15]` 网格。每次达成五连后，调用 `markWinCellsAsScored()` 沿胜利线逐格标记。后续扫描遇到 `scoredWin == true` 的格子立即停止计数，确保"同一组五连只计一次"。

**ChessFrame 集棋子获胜制**：每次达成五连/六连不在立即判胜，而是：
1. 按归属在上方/下方 ChessFrame 从左至右放置一枚棋子
2. PVE 模式：上方 = 玩家方，下方 = AI 方
3. PVP 模式：上方 = 黑方，下方 = 白方
4. 率先集满 5 枚者获胜
5. 棋子放置伴随完整的碎片聚合 + 飞行动画

### 4.4 AI 决策系统

**三级难度**：
- **初级**：随机落子，不评估棋盘局势
- **明智**：使用启发式评估函数，考虑连子长度、开放端、威胁度
- **大神**：更敏感的评估阈值，AI 卡牌出牌更积极

**卡牌决策**：`evaluateCard()` 函数根据当前棋盘状态和手牌进行启发式评分。例如：
- 隐忍卡：基础评分 80，若对手棋子 ≥8 颗则额外加 40 分
- 紫卡优先：AI 检测到持有紫卡时优先出牌，体现间接博弈策略

**AI 动画**：AI 出牌拥有完整的动画流程（Portal 生成 → 展示 → 滑入 Reader → 湮灭 → 效果触发），与玩家出牌保持视觉一致性，通过 `AICardPlayState` 状态机驱动。

### 4.5 音乐系统

采用**双音轨串行过渡**设计，避免两首战斗音乐重叠：
- `battle_01`：默认战斗音乐，音量 40
- `battle_02`：CardReader/Portal 触发时自动切换，音量 40
- 过渡机制：1s 淡出 → stop → 1s 淡入
- 四子连星触发 `battle_02` 的 30s 危机叠加延长（无上限）
- 胜利/失败播放对应结算音乐（`win.ogg` / `fail.ogg`）

## 七、总结与展望

### 7.1 项目总结

本项目成功实现了一款融合五子棋与卡牌玩法的策略游戏，完成了从需求分析、系统设计到编码实现、测试验证的完整软件开发流程。主要成果包括：

1. **创新的玩法融合**：通过行动点系统将棋类与卡牌有机统一，创造了"每一步都在下棋和出牌之间权衡"的策略深度
2. **完整的卡牌体系**：9 张差异化卡牌（6 橙 + 2 紫 + 1 新橙），涵盖直接效果、延迟效果、诅咒传送等多种机制
3. **高性能碎片动画**：基于预计算缓存的粒子系统，支持棋子和卡牌的生成、销毁、转化、湮灭、聚合全流程，视觉效果丰富且性能稳定，是本项目最大的技术亮点
4. **健全的 AI 系统**：三级难度、启发式评估、完整动画流程
5. **良好的用户体验**：2K 分辨率适配、界面过渡动画、暂停菜单、详细的文档和开发日志

### 7.2 技术反思

- **预计算缓存的优势**：碎片动画的预计算策略在保证视觉随机性的同时大幅减少运行时开销，这一模式在游戏开发中值得推广
- **状态机驱动的可行性**：整个游戏通过状态机管理（游戏状态 + 展示状态 + 动画阶段），虽然分支较多但逻辑清晰可维护
- **模块化设计的局限**：GameEngine 类体积较大（~6000 行），未来可考虑拆分渲染模块和动画模块为独立类

### 7.3 未来展望

- JSON 配置文件加载牌库，支持自定义卡牌
- 更多紫卡品种，丰富博弈维度
- 网络对战支持
- 棋子落下音效与卡牌音效的进一步补全
- 移动端适配
